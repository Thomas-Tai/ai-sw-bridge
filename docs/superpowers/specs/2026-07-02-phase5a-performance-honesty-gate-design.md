# Phase 5A — The Performance Honesty Gate (Design Specification)

> **Status:** DRAFT for review gate · **Date:** 2026-07-02 · **Author:** (SDD orchestrator)
> **Governing plan:** `docs/superpowers/specs/2026-07-01-commercial-google-standard-elevation-design.md` §11 (Phase "Expandability", **decomposed** — Phase 5A = perf gate, Phase 5B = installer, per the ratified segregation).
> **Predecessors:** [[project_phase4_i18n_honesty_gate_shipped]] (the Model-B pattern this extends), the v0.11 SLI instrumentation (`tools/regression_check.py`, commits `3019eac`/`0dffa96`).

---

## 1. Objective

Give `ai-sw-bridge` a **performance regression gate that is real in CI** — despite CI having no SOLIDWORKS seat and therefore being unable to *measure* performance. We do this by treating the codebase's measured performance as a **verified fact that must be documented on a live seat**, and using a seatless CI test to make it **structurally impossible for the code to silently outgrow that documentation**. Honest lag (the hot path changed, not yet re-measured) is permitted and *visible*; silent perf rot is forbidden.

This is the **Model-B honesty architecture from Phase 4, extended from the i18n domain into the performance domain**: CI cannot do the expensive thing (translate / run SOLIDWORKS), so it instead enforces a biconditional over a committed artifact — *the measurement is stale ⇔ it says so*.

Two deliverables, one atomic phase:
1. **The Perf Receipt** — a git-anchored, machine-readable JSON artifact recording the last live-seat performance measurement, generated by extending the *existing* `regression_check.py` tooling.
2. **The Honesty Gate** — `tests/test_perf_receipt.py`, a seatless CI test enforcing that the receipt is present, well-formed, **derived-green when current**, and **stale ⇔ lag-acknowledged**.

## 2. Ratified anchors (from the brainstorm gate)

Settled in dialogue; the review gate does **not** reopen these absent new technical evidence:

- **A2.1 — The metric is the end-to-end feature build.** Not COM-marshalling latency, not CLI cold-start. The thing operators feel is `ai-sw-build` turning a `spec.json` into geometry on a live seat. `regression_check.py` already measures exactly this (wall-clock per `examples/*/spec.json` build, p50/p95/p99), so the metric is *already chosen and already implemented*.
- **A2.2 — The ceiling is the SLO that already shipped.** SLO-01 `p95 < 12 s`, SLO-02 `p99 < 25 s`, plus baseline-regression trips at `p95 +15%` / `p99 +25%` vs the checked-in baseline (`tools/perf_baselines/v0.10.json`). We wire the thresholds v0.11 already encoded; we do **not** invent new ones.
- **A2.3 — Single tier, no seatless CI micro-benchmark.** A wall-clock cold-start/dry-run timer running live in GitHub Actions was explicitly rejected: shared multi-tenant runners make timing gates flappy, and a flappy gate gets disabled. The receipt honesty-gate is the sole CI-visible perf signal, and it is *deterministic* (it reads committed numbers and does arithmetic).
- **A2.4 — There is no PR gate; the enforcing surface is push-to-master CI.** This repo ships solo by fast-forward. "Block the PR" is theater. The gate runs as a normal pytest test on every push, exactly like `test_i18n_staleness.py`. (This *corrects* a stale assumption in `regression_check.py`'s own docstring, which references a `perf-baseline-bump` **PR label** — machinery that never existed here.)

## 3. Current state (measured 2026-07-02)

### 3.1 What exists
- **`tools/regression_check.py`** (v0.11) — builds every `examples/*/spec.json` on a live seat in `--no-dim --verify-mass` mode, records golden volume + per-feature deltas, captures wall-clock per build, computes p50/p95/p99, and enforces SLO-01/02 + baseline regression via `--baseline-compare`. Exit 0 pass / 1 regression / 2 build error. **It is fully built and orphaned** — invoked by nothing in `.github/workflows/`.
- **`tools/perf_baselines/v0.10.json`** — a real baseline captured on a live seat (`p50 5.985`, `p95 11.933`, `p99 12.537`, `n_specs 15`, Windows + Python 3.10). Read by `regression_check.py --baseline-compare`.
- **20 example specs** under `examples/*/spec.json` — the timed corpus.
- **CI `fetch-depth: 0`** on the `test` job (landed in Phase 4 for the i18n gate) — the perf gate's git-history dependency is therefore **already satisfied**; no new CI change is required to enforce it.

### 3.2 What is missing
- **No receipt.** `regression_check.py` can write a *baseline* (`--write-baseline`) but not a **git-anchored receipt** binding a measurement to the commit it measured.
- **No gate.** Nothing enforces that a passing measurement exists for the current hot path. The SLO is enforceable only if a human remembers to run the orphaned script.
- **No freshness notion.** Nothing detects that the build engine changed since the last measurement.

## 4. Scope

### 4.1 In scope
- Extend `regression_check.py` with `--emit-receipt <path>`: after a `--check --baseline-compare` run, write the Perf Receipt JSON (§5.2), stamping the current `git rev-parse HEAD` as `measured_at` and echoing the raw percentiles + baseline reference.
- Build `tests/test_perf_receipt.py` — the seatless honesty gate (§5.3–§5.5).
- Generate and commit the initial `tools/perf_baselines/receipt.json` on the live seat (the one deliberate seat exercise of this phase).
- Document the ritual + baseline-bump discipline in `CONTRIBUTING.md` and a short `docs/perf_gate.md`.

### 4.2 Out of scope (non-goals — named so the omission is intentional)
- **Seatless CI micro-benchmarks** (A2.3, rejected).
- **Measuring performance in CI** — impossible without a seat; the gate is a *documentation-integrity* check, not a timer.
- **Cryptographic anti-forgery.** This is an *honesty* gate for an honest solo maintainer, not a fraud-proof one (zero budget, no signing). The trust model is stated in §5.6; it prevents *forgetting*, not *lying* — identical in spirit to i18n's `translated-from`.
- **Changing the measurement logic** of `regression_check.py` — only the additive `--emit-receipt` path is new.
- **New SLO thresholds** (A2.2 reuses the shipped ones).

## 5. The Honesty Gate — design

### 5.1 The two clauses
The gate enforces a **freshness biconditional** and a **green-when-current** clause over a single committed receipt.

Let `sha0 = receipt.measured_at`, and let **PERF_SURFACE** be the feature-build hot path:
- `src/ai_sw_bridge/spec/` (builder + handlers + schema + validator + sketches)
- `src/ai_sw_bridge/features/` (the `feature_add` registry)
- `src/ai_sw_bridge/cli/build.py` (the entrypoint `regression_check.py` drives)
- `examples/*/spec.json` (the timed corpus inputs — **not** the generated `golden*.json` outputs)

Definitions:
- **stale(receipt)** ≝ `git rev-list <sha0>..HEAD -- <PERF_SURFACE>` is non-empty (the hot path advanced since the measurement).
- **lag_acknowledged** ≝ the receipt's `lag_acknowledged` field is `true`.

**Clause 1 — freshness biconditional:** `stale ⇔ lag_acknowledged`.
- stale ∧ ¬acknowledged → **FAIL** (silent perf rot — the exact thing this phase prevents).
- fresh ∧ acknowledged → **FAIL** (crying wolf — a stale-lag flag left on a current measurement).
- stale ∧ acknowledged → PASS (honest lag — permitted, and visibly committed).
- fresh ∧ ¬acknowledged → PASS (current and truthful).

**Clause 2 — green-when-current:** if `¬lag_acknowledged` (the receipt claims to be current), CI **re-derives** the verdict from the receipt's own raw percentiles + the referenced baseline + the canonical thresholds imported from `regression_check.py`, and requires it to pass:
- `p95 < SLO_01_P95_MAX_S` and `p99 < SLO_02_P99_MAX_S`, **and**
- `p95 ≤ baseline.p95 × (1 + BASELINE_P95_REGRESSION_PCT)` and `p99 ≤ baseline.p99 × (1 + BASELINE_P99_REGRESSION_PCT)`.

When `lag_acknowledged` is true, Clause 2 **and the corpus-size check (§5.3 test 6) are not applied** — the numbers are admittedly stale and the corpus may have grown; the visible lag flag *is* the debt. Re-measuring at a fresh SHA flips `lag_acknowledged` back to false and re-arms both. The structural checks (§5.3 tests 1–3) and the freshness biconditional (Clause 1) apply **always**, lag or not.

> **Key property:** CI never trusts a committed pass/fail boolean. It recomputes the verdict from raw numbers against thresholds it imports from source. A hand-edited `"verdict": "pass"` cannot fool it — only faking the raw `p95`/`p99` (i.e. lying about the measurement) could, which is unpreventable without a CI seat and out of the honesty-gate's threat model (§5.6).

### 5.2 The Perf Receipt data contract
Committed at `tools/perf_baselines/receipt.json`. The receipt carries **raw measurements only** — no committed verdict; the verdict is CI-derived.

```json
{
  "schema_version": 1,
  "measured_at": "204959f8be396499588b89aacdbed7227503dee9",
  "baseline": "tools/perf_baselines/v0.10.json",
  "p50": 5.985,
  "p95": 11.933,
  "p99": 12.537,
  "n_specs": 20,
  "spec_times": [4.375, 4.703, "..."],
  "host_meta": {
    "platform": "Windows-11-10.0.26200-SP0",
    "python": "3.12.x",
    "sw_revision": "32.1.0"
  },
  "timestamp": "2026-07-02T10:00:00+00:00",
  "lag_acknowledged": false,
  "lag_reason": null
}
```

| Field | Type | Meaning / gate use |
|---|---|---|
| `schema_version` | int | Forward-compat pin; gate asserts `== 1`. |
| `measured_at` | str (40-hex) | Git SHA of HEAD at measurement time — the freshness anchor (Clause 1). Must be reachable. |
| `baseline` | str (repo-relative path) | The baseline `p95`/`p99` are compared against (Clause 2). Must exist on disk and carry numeric `p95`/`p99`. |
| `p50` / `p95` / `p99` | float (seconds) | Raw measured percentiles. Clause 2 re-derives the verdict from `p95`/`p99`. |
| `n_specs` | int | Corpus size measured (audit; sanity vs disk). |
| `spec_times` | list[float] | Per-spec wall-clock (audit trail). |
| `host_meta` | object | `platform`, `python`, best-effort `sw_revision` — provenance of the measurement. |
| `timestamp` | str (ISO-8601) | Human provenance. Not gate-load-bearing. |
| `lag_acknowledged` | bool | The honest-lag escape hatch (Clause 1). |
| `lag_reason` | str \| null | Required non-empty **iff** `lag_acknowledged` is true (why not re-measured). |

### 5.3 The tests (`tests/test_perf_receipt.py`)
1. `test_receipt_exists_and_wellformed` — file present; `schema_version == 1`; required fields present + typed; `measured_at` matches `[0-9a-f]{7,40}`; if `lag_acknowledged` then `lag_reason` non-empty.
2. `test_measured_at_reachable` — `git cat-file -e <measured_at>` succeeds (rebase/force-push orphan or forged SHA → FAIL).
3. `test_baseline_reference_resolves` — `receipt.baseline` exists on disk and has numeric `p95`/`p99`.
4. `test_freshness_biconditional` — Clause 1: `stale ⇔ lag_acknowledged`.
5. `test_green_when_current` — Clause 2: if `¬lag_acknowledged`, the CI-derived SLO + regression verdict (thresholds imported from `regression_check.py`) passes.
6. `test_n_specs_matches_corpus` — `receipt.n_specs == len(glob("examples/*/spec.json"))` (a shrunk/forgotten corpus can't masquerade as a full measurement). **Suspended under `lag_acknowledged`** (a grown corpus is expected honest debt), same as Clause 2.

### 5.4 Fail-loud edges (mirrors i18n §5.4)
- Missing / malformed receipt → FAIL.
- Blank / non-hex `measured_at`, or `lag_acknowledged` without `lag_reason` → FAIL.
- `measured_at` not in git history → FAIL.
- `baseline` path missing, or missing `p95`/`p99` → FAIL.
- Non-numeric / missing percentiles → FAIL.

### 5.5 Git-history dependency & local ergonomics
The gate shells `git rev-list` / `cat-file` (like `test_i18n_staleness.py`). It requires history for PERF_SURFACE + `measured_at` reachability.
- **CI:** the `test` job already checks out `fetch-depth: 0` (Phase 4). **No CI change needed** — the test auto-collects into the seat-safe suite and enforces for real.
- **Local / shallow:** if `git` is unavailable or `measured_at` is unreachable in a shallow clone, `pytest.mark.skipif` skips **with an explicit reason** (never a false pass, never a false block). CI is the enforcing surface.

### 5.6 Trust model (explicit)
The receipt is hand-committable JSON with no signature (zero budget). The gate is designed to defeat the **failure mode that actually happens to a diligent solo maintainer: forgetting to re-measure after touching the build engine.** It does not defend against a maintainer who deliberately fabricates `p95`. This is the same trust posture as i18n's `translated-from` and is stated here so the boundary is not mistaken for a hole. CI *re-deriving* the verdict from raw numbers (§5.1) already removes the easy forgery (flipping a boolean); only fabricating the raw measurement remains, which is conscious dishonesty, not an oversight.

## 6. CLI mechanics — generating the receipt (the seat ritual)

Receipt generation is a **deliberate, maintainer-initiated live-seat run** — the one seat exercise of Phase 5A, analogous to the Phase-3 seat re-fire. It is non-destructive (it builds example specs into fresh blank parts, reads mass, never overwrites operator files), but it *does* drive a live seat, so it is **not** part of the automated suite and never runs in CI.

New flag on the existing tool:

```bash
# On the operator workstation, with a live SOLIDWORKS seat:
python tools/regression_check.py \
    --check \
    --baseline-compare tools/perf_baselines/v0.10.json \
    --emit-receipt tools/perf_baselines/receipt.json
```

`--emit-receipt` writes the §5.2 JSON after the run: it captures `git rev-parse HEAD` as `measured_at`, the run's raw `p50/p95/p99` + `spec_times`, `n_specs`, `host_meta` (platform/python, best-effort `sw_revision` from the build result), a UTC `timestamp`, and `lag_acknowledged: false` (a just-measured receipt is current by construction). The maintainer then commits `receipt.json`.

**Honest-lag path (no seat available, hot path changed):** the maintainer hand-sets `lag_acknowledged: true` + a `lag_reason` on the existing receipt and commits — CI stays green (Clause 1: stale ∧ acknowledged), the debt is visible and greppable, and Clause 2 is suspended until the next real measurement.

### 6.1 Baseline-bump discipline
When perf legitimately and acceptably regresses (e.g. a new inherently-slower feature), the maintainer **bumps the baseline** rather than lying to the gate:

```bash
python tools/regression_check.py --check \
    --baseline-compare tools/perf_baselines/v0.10.json \
    --write-baseline  tools/perf_baselines/v1.7.json \
    --emit-receipt    tools/perf_baselines/receipt.json
```

then repoints `receipt.baseline` to the new file. In a solo FF repo, "reviewed baseline bump" = **a deliberate, self-documented baseline commit** (superseding the docstring's never-built `perf-baseline-bump` PR-label notion). The old baseline is retained for provenance.

## 7. Definition of Done

- [ ] `regression_check.py` gains `--emit-receipt <path>` (additive; existing modes unchanged); black/flake8/mypy-clean.
- [ ] `tools/perf_baselines/receipt.json` generated on the live seat, committed, **fresh + derived-green** (`lag_acknowledged: false`).
- [ ] `tests/test_perf_receipt.py` green: well-formed + reachable `measured_at` + baseline resolves + freshness biconditional + green-when-current + corpus-size.
- [ ] Gate **bite-proven** (each reverted): (a) touch a PERF_SURFACE file after `measured_at` with no lag flag → FAIL; (b) `lag_acknowledged: true` on a fresh receipt → FAIL; (c) blank/unreachable `measured_at` → FAIL; (d) fresh receipt with `p95` above SLO and no lag → FAIL (Clause 2); (e) `baseline` pointing at a missing file → FAIL.
- [ ] Seat-safe suite green (baseline + the new perf-receipt tests); live seat untouched by the *suite* (the receipt *generation* is the one deliberate, non-destructive seat run, maintainer-initiated).
- [ ] `CONTRIBUTING.md` + `docs/perf_gate.md` document the receipt ritual, the honesty biconditional, and the baseline-bump discipline.
- [ ] No CI change required beyond the test auto-collecting (fetch-depth:0 already present) — verified.
- [ ] isPrivate-guarded FF push to master; branch `docs/commercial-elevation` only.

## 8. Risks & mitigations

| Risk | Mitigation |
|---|---|
| **CI can't measure perf** (no seat) | By design: the gate checks *documentation integrity*, not live timing. The measurement is a seat ritual; CI enforces freshness + derived-green deterministically. |
| **Forged receipt** | CI re-derives the verdict from raw numbers (never trusts booleans); §5.6 trust model bounds the residual to conscious fabrication, out of scope. |
| **Flappy CI perf test** | Eliminated at the root — the CI test does *no timing*; it reads committed numbers + does arithmetic + `git rev-list`. Deterministic. |
| **Permanent `lag_acknowledged`** dodging the SLO | Visible, greppable honest debt (identical to a permanently-bannered stale i18n mirror); Model-B tolerates honest lag by design. A future ratchet could bound lag age; out of scope now. |
| **PERF_SURFACE too broad/narrow** | Scoped to the build engine + `examples/*/spec.json` inputs (excludes generated `golden*.json`), so ordinary doc/test commits don't trip freshness, but any real hot-path change does. |
| **Receipt generation drives the live seat** | Non-destructive (own blank parts, no saves over operator files); maintainer-initiated, never in the suite or CI; PID-unchanged invariant still applies to the *automated* suite. |

## 9. Open decisions for the review gate

### 9.1 Receipt location
Proposed `tools/perf_baselines/receipt.json` (co-located with the baselines it references). Alternative: a top-level `perf_receipt.json` or `docs/perf/receipt.json`. **Recommendation:** keep it in `tools/perf_baselines/` — discoverable next to the baselines, and the whole directory is already the perf artifacts' home. (Ratify or override.)

### 9.2 `lag_acknowledged` as a JSON field vs a sentinel comment
i18n used an HTML-comment sentinel because mirrors are Markdown. The receipt is JSON, so a boolean field (`lag_acknowledged` + `lag_reason`) is the natural, parseable equivalent. **Recommendation:** the JSON field pair. (Ratify or override.)

### 9.3 Whether Clause 2 should also run under acknowledged lag
Considered enforcing derived-green even on acknowledged-lag receipts (against their stale numbers). **Recommendation:** do **not** — stale numbers shouldn't gate a green/red gate on current SLO; the visible lag flag is the honest signal, and re-measurement re-arms Clause 2. (Ratify or override.)

## 11. Success criterion (one sentence)

After Phase 5A, the codebase's end-to-end build performance is a committed, seat-measured fact, and it is **structurally impossible for the feature-build engine to regress past its SLO — or to drift away from its last measurement — without CI going red or the receipt visibly, greppably declaring itself lagged**, all enforced deterministically in a seatless runner at zero flakiness.
